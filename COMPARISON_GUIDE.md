# 두 버전 비교 가이드

Matt Pocock의 "Software Fundamentals" 원칙 + Neo의 형이상학/신앙적 framework을 담은 스킬을
두 가지 형식으로 작성했습니다. 검토 후 채택할 버전을 선택해주세요.

---

## V1: Monolith (단일 파일)

**구조:**
```
v1-monolith/
└── SKILL.md   (한 파일에 모든 내용)
```

**장점:**
- 한 번에 다 읽힘 (참조 파일 로드 불필요)
- 짧은 스킬, 빠른 발동에 유리
- AI가 매번 전체 컨텍스트를 다 봄

**단점:**
- 파일이 길어짐 (현재 ~280줄, 한계 500줄에 여유 있긴 함)
- 깊이 있는 참조 자료를 다 펼치기 어려움
- 형이상학 / 6가지 함정 / Grill Me 질문 / Deep Module 판별 모두 한 문서에 압축 → 각 항목 깊이 제한

**적합한 경우:**
- 모든 코딩에서 가볍게 발동시키고 싶을 때
- 빠른 의사결정용 체크리스트가 필요할 때
- 스킬 시스템에 익숙하지 않은 환경 (단일 파일이 관리 쉬움)

## V2: Bundle (코어 + 참조 문서)

**구조:**
```
v2-bundle/
├── SKILL.md                          (코어: 트리거 + 7단계 워크플로우)
└── references/
    ├── PHILOSOPHY.md                 (형이상학 + 신앙적 근본)
    ├── SIX_PITFALLS.md               (6가지 함정 상세 + 안티 패턴)
    ├── GRILL_ME_PROTOCOL.md          (의도 추궁 질문 템플릿)
    └── DEEP_MODULE_REVIEW.md         (Deep/Shallow 판별 체크리스트)
```

**장점:**
- 코어 SKILL.md는 짧고 명료 (~95줄) → 항상 컨텍스트에 있어도 부담 없음
- 깊이 있는 참조 자료를 필요할 때만 로드 (Progressive Disclosure)
- 각 참조 문서가 독립적이라 단독으로도 활용 가능
- 형이상학 / 6가지 함정 / Grill Me / Deep Module 각각 충분한 깊이로 작성
- 확장이 쉬움 (새로운 references/ 추가만 하면 됨)

**단점:**
- 파일이 여러 개 → 관리 복잡도 약간 증가
- AI가 작업 중 참조 파일을 안 읽고 코어만 보고 끝낼 위험 있음 (트리거 규칙이 잘 작동해야 함)
- 처음 셋업 시 진입 장벽 약간 있음

**적합한 경우:**
- 진지하게 모든 코딩 시 깊은 검토를 원할 때
- 각 layer(형이상학/함정/질문/모듈)를 깊이 있게 펼치고 싶을 때
- 추후 새 원칙 추가 가능성 있을 때
- Claude Code의 progressive disclosure 메커니즘을 활용하고 싶을 때

## 핵심 차이 요약

| 항목 | V1 Monolith | V2 Bundle |
|------|-------------|-----------|
| 파일 수 | 1개 | 5개 |
| 코어 길이 | ~280줄 | ~95줄 |
| 총 분량 | ~280줄 | ~700줄 (코어+참조 합계) |
| 깊이 | 압축됨 | 각 주제별 충분히 깊음 |
| 발동 부담 | 작음 | 작음 (코어만 항상 로드) |
| 확장성 | 제한적 | 우수 |
| 관리 복잡도 | 단순 | 약간 복잡 |
| 형이상학 layer | 1단원으로 압축 | 독립 문서로 깊이 있게 |
| 신앙적 근본 | 인용구로 명시 | 독립 문서에서 신학적 함의까지 |

## 제 추천 (참고용)

**V2 Bundle을 추천드립니다.** 이유:

1. Neo님이 다층 사유(엔지니어링 + 형이상학 + 신앙)를 통합하길 원하시는데, V1은 그 깊이를 다 못 담음.
2. 진행 중인 프로젝트가 여러 개(TTL 맞고, HOSU, TTL Forge 등)라 각 참조 문서에 프로젝트별 적용 예시 추가가 쉬움.
3. Pocock의 GitHub Skills 저장소 패턴도 참조 문서 분리 방식.
4. Claude Code의 progressive disclosure를 제대로 활용 가능.

단, V1이 더 나은 점도 분명히 있습니다 — 한눈에 다 보이는 안심감과 단순함. 검토 후 Neo님 직관에 맞는 쪽을 선택하시면 됩니다.

## 검토 시 점검 포인트

각 버전 검토 시 다음을 확인해보세요:

### 양쪽 공통

1. **트리거 description이 충분히 푸시하는가?** (Anthropic 가이드: "undertrigger 경향이 있으므로 약간 푸시"
2. **TTL 프로젝트 적용 예시가 실제 작업에 도움되는가?**
3. **신앙적 근본(잠언 1:7, 창세기 1:1)이 자연스럽게 통합되었는가, 어색한가?**
4. **형이상학 layer(a~z → ㄱ~ㅎ)가 실제 코딩 의사결정에 도움이 되는가?**

### V1 검토 시

- 한 파일에 다 담겨서 답답하지 않은가?
- 빠뜨려도 될 부분이 있는가?

### V2 검토 시

- 참조 문서가 너무 많지 않은가? 합칠 만한 게 있나?
- 코어 SKILL.md만 봐도 핵심이 전달되는가?
- 참조 문서 간 중복이 너무 심하지 않은가?
