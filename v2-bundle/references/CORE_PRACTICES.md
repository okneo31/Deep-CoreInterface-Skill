# CORE PRACTICES: 지켜야 할 네 가지 긍정 원칙

> "AI coding tools are overhyped and powerful at the same time... The devs who succeed aren't the ones who delegate everything or nothing. They're the ones who fall back on engineering fundamentals... **decades-old ideas that didn't break. They got more important.**" — Matt Pocock

`SIX_PITFALLS.md`가 **피해야 할 6가지 함정**을 다룬다면, 이 문서는 **지켜야 할 4가지 긍정 원칙**을 다룬다.
함정 회피만으로는 부족하다. 적극적인 실천 원칙이 같이 가야 한다.

Pocock이 18개월간 AI 에이전트로 개발자들을 가르치며 검증한 결과 — 성공하는 개발자는 "모든 걸 위임"하지도 "전혀 위임 안 하지도" 않는 사람. **엔지니어링 펀더멘털로 되돌아가는** 사람이었다. 그 펀더멘털이 이 네 가지다.

## 1. Ubiquitous Language (편재적 언어)

> 📖 **창세기 11:6** (바벨탑 직전)
>
> "여호와께서 이르시되 이 무리가 한 족속이요 **언어도 하나이므로** 이같이 시작하였으니 이후로는 그 경영하는 일을 금지할 수 없으리로다 하시고"
>
> 언어가 통일되어 있을 때 사람은 무엇이든 시작할 수 있다. 비즈니스가 쓰는 용어, 코드가 쓰는 용어, 사용자가 쓰는 용어, 회의에서 쓰는 용어 — 다 같은 단어를 가리키게 하라. 언어가 흩어진 곳에서는 어떤 일도 끝까지 가지 못한다.

### 출처
Eric Evans, "Domain-Driven Design" (DDD). 비즈니스 도메인 용어와 코드 용어를 통일하는 원칙.

### 핵심 원리
- **번역 비용 = 버그.** 사용자가 "주문 취소"라 부르는 걸 코드에선 `cancelOrder`, UI에선 "결제 해지", 데이터베이스에선 `tx_void`로 쓰면 — 그 번역 사이 어딘가에서 반드시 누수가 일어난다.
- **하나의 단어, 하나의 의미.** 같은 단어가 컨텍스트마다 다른 걸 가리키면(예: "사용자" — 결제 시점엔 구매자, 게임 안에선 플레이어, 관리자 화면에선 회원) 그 경계를 명시해야 한다.
- **언어는 발견되는 것이지 발명하는 것이 아니다.** 사용자/도메인 전문가가 이미 쓰는 단어가 있다면 그것을 쓴다. 기술자가 새 용어를 만들어 강요하지 말 것.

### 적용 방법
- 새 프로젝트 시작 시 **용어집**(Glossary)을 먼저 만든다. 한 화면이 아니라 별도 문서.
- 코드의 클래스/함수 이름, UI 라벨, 데이터베이스 컬럼명, 문서 — 모두 그 용어집을 따른다.
- 새 용어가 등장하면 용어집에 먼저 등재한 후 코드에 반영.

### TTL 프로젝트 적용 예시
- **TTL 맞고:** "고스톱" / "맞고" / "치다" / "낸다" / "쪼다" 같은 한국 화투 고유 용어를 코드에 그대로 반영. 영어로 번역하지 말 것 (`playCard`보다 `nadaCard` 또는 `naeda` 식).
- **HOSU:** "관리비" / "거주자" / "분쟁" / "공동구매" 같은 단어를 사용자/관리사무소/입주민 모두 같은 의미로 쓰도록 합의. VibeScore는 "분위기 점수"인지 "공동체 점수"인지 명확히.
- **줍딜:** "시세" / "정상가" / "급매" 같은 중고시장 용어를 표준화. 사용자가 부르는 그대로 코드에 박기.

### 안티 패턴
- 같은 개념을 코드 / UI / 문서에서 각기 다른 단어로 부름.
- 사용자가 부르는 용어를 무시하고 기술 용어로 강제 ("결제" 대신 `transaction`).
- 한 단어가 컨텍스트마다 다른 의미인데 그 경계를 명시 안 함.

## 2. Vertical Slices (수직 슬라이스)

> 📖 **빌립보서 3:13-14**
>
> "형제들아 나는 아직 내가 잡은 줄로 여기지 아니하고 **오직 한 일 즉 뒤에 있는 것은 잊어버리고 앞에 있는 것을 잡으려고 푯대를 향하여** 그리스도 예수 안에서 하나님이 위에서 부르신 부름의 상을 위하여 좇아가노라"
>
> 한 번에 한 슬라이스. 시작에서 끝(푯대)까지 한 번에 가는 작은 조각 하나. 모든 것을 동시에 50%씩 하는 게 아니라, 한 슬라이스를 100% 끝낸다.

### 출처
XP(Extreme Programming) / Agile 운동. "벽돌을 한 층씩 쌓지 말고, 작은 집을 먼저 짓고 그 위에 더 짓는다"는 비유로도 표현됨.

### 핵심 원리
- **수평 레이어 ✗:** UI 다 만들고 → 로직 다 만들고 → DB 다 만드는 식. 마지막에야 통합되고, 그제야 문제 발견. 너무 늦음.
- **수직 슬라이스 ✓:** 한 기능 하나를 UI부터 DB까지 처음부터 끝까지 동작하게 만든 다음, 다음 기능으로. 매 슬라이스가 끝나면 출시 가능 상태.
- **YAGNI(You Aren't Gonna Need It)와 통함.** 미리 만들어두지 말고, 슬라이스에 필요한 만큼만.

### 적용 방법
- 새 기능 기획 시 **가장 얇은 슬라이스**부터 정의. "로그인 → 대시보드 → 결제" 한 흐름 전체를 가장 단순한 형태로 먼저.
- 슬라이스 완료 = UI + 비즈니스 로직 + 데이터 + 테스트 모두 동작.
- 슬라이스가 출시 가능한 상태가 되면 다음 슬라이스로.

### TTL 프로젝트 적용 예시
- **TTL 맞고:** 첫 슬라이스 = "혼자 카드 한 장 내고 점수 계산되는 화면 하나"가 끝까지 동작. AI 코칭, 음성, 결제는 후속 슬라이스.
- **HOSU:** 첫 슬라이스 = "거주자 한 명이 관리비 확인하는 흐름 하나"가 끝까지 동작. VibeScore, 분쟁 중재는 후속.
- **줍딜:** 첫 슬라이스 = "한 플랫폼(예: 번개장터) 검색해서 시세 보여주는 흐름"이 끝까지 동작. 다른 플랫폼/프리미엄 기능은 후속.

### 안티 패턴
- "DB부터 다 설계해두자" → 6개월 후에야 UI 보임 → 그제야 비즈니스가 "이건 다른데?"
- "프론트는 다 했는데 백엔드가 안 됨" → 모자이크 같은 상태.
- 슬라이스가 너무 두꺼움 (예: "결제 시스템 전부" — 슬라이스가 아니라 단계).

## 3. TDD (Test-Driven Development, 테스트 주도 개발)

> 📖 **데살로니가전서 5:21**
>
> "**범사에 헤아려 좋은 것을 취하고**"
>
> 헤아림(테스트) 후에 취함(채택). 헤아림 없이 취하지 말 것. AI가 짠 코드든 사람이 짠 코드든 마찬가지다. 검증 절차가 먼저고, 채택은 그 다음.

### 출처
Kent Beck, "Test-Driven Development: By Example". Red-Green-Refactor 사이클로 유명.

### 핵심 원리
- **Red:** 실패하는 테스트를 먼저 쓴다. (아직 코드 없으니 당연히 실패)
- **Green:** 그 테스트를 통과하는 최소한의 코드를 짠다.
- **Refactor:** 테스트가 통과한 상태를 유지하며 코드를 정돈한다.
- **테스트는 명세다.** 테스트가 비즈니스 요구사항을 가장 정확히 표현한다. 자연어 명세보다 더 엄격하고 검증 가능.

### AI 시대에 특히 중요한 이유
- AI가 짠 코드를 검증할 표준이 필요하다. 사람이 매번 눈으로 검토하는 건 불가능.
- 테스트가 있으면 AI에게 "이 테스트를 통과하게 짜줘"라고 명확한 명세를 줄 수 있다.
- 테스트가 없으면 AI가 만든 코드가 "그럴듯하지만 실제로는 작동 안 함" 상태로 누적됨.

### 적용 방법
- 새 기능 시작 시 **테스트부터 작성**. "성공 시 X 반환, 실패 시 Y 던짐"을 코드로 박는다.
- AI에게 코드 짜라고 할 때 **테스트와 함께 던진다**: "이 테스트를 통과하도록 구현해줘."
- 매 슬라이스(§2)에 테스트 포함. 테스트 없는 슬라이스 = 완료 아님.

### TTL 프로젝트 적용 예시
- **TTL 맞고:** 룰 엔진 핵심부터 TDD. "이 카드 조합이 들어오면 이 점수가 나와야 함" 테스트 수십 개 먼저, 그 다음 룰 엔진 구현.
- **HOSU:** VibeScore 계산식부터 TDD. "이 데이터가 들어오면 이 점수가 나와야 함" 테스트 먼저.
- **줍딜:** 시세 계산 알고리즘부터 TDD. "이 매물 데이터가 들어오면 시세가 N원이어야 함" 테스트 먼저.

### 안티 패턴
- "급하니까 일단 짜고 테스트는 나중에" → "나중에"는 영원히 안 옴.
- 테스트가 있긴 한데 구현 후에 작성됨 (구현에 맞춰 쓴 테스트는 명세가 아님).
- AI에게 코드만 시키고 테스트는 안 시킴 → AI가 만든 코드가 검증 안 됨.

## 네 원칙의 통합

이 네 가지는 따로따로 적용하는 게 아니라 **함께** 작동한다.

```
[Ubiquitous Language로 사용자 의도를 같은 언어로 잠그고]
            ↓
[Vertical Slice 단위로 작업 범위를 정한 다음]
            ↓
[TDD로 그 슬라이스의 명세를 테스트로 박고]
            ↓
[Deep Module 구조로 구현한다]
```

하나라도 빠지면 다른 셋이 약해진다:
- **언어가 통일 안 되면** — 슬라이스 정의가 흐려지고 테스트가 무엇을 검증하는지 모호해짐.
- **슬라이스 단위 없으면** — 테스트가 너무 거대해지거나 너무 잘게 쪼개짐.
- **TDD 없으면** — Deep Module이 실제로 깊은지 검증할 길이 없음.
- **Deep Module 없으면** — 테스트 가능한 단위가 안 나옴.

네 원칙은 한 시스템이다.

## 매주 자가 점검

매주 월요일 30분, 진행 중인 프로젝트 1개에 대해:

1. **Ubiquitous Language:** 이번 주에 쓴 코드/문서/UI 라벨에 용어집과 일치하지 않는 단어가 있나?
2. **Vertical Slices:** 이번 주에 만든 것이 끝에서 끝까지 동작하는가, 아니면 중간만 있나?
3. **TDD:** 이번 주에 짠 코드 중 테스트 없이 들어간 게 있나?
4. **Deep Modules:** 이번 주에 만든 모듈 중 Shallow인 게 있나? (→ `DEEP_MODULE_REVIEW.md`의 5가지 질문)

하나라도 No가 아니면, 그게 다음 주 첫 작업이다.
