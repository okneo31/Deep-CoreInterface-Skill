# SIX PITFALLS: 여섯 가지 함정

> 📖 **잠언 22:3**
>
> "**슬기로운 자는 재앙을 보면 숨어 피하여도 어리석은 자는 나가다가 해를 받느니라**"
>
> 함정은 알아보는 사람만 피할 수 있다. 이 문서는 AI 시대에 좋은 코드를 망치는 6가지 함정을 미리 알아보기 위함이다. 알아본 자만 비켜선다.

Matt Pocock의 41만 뷰 발표 "Software Fundamentals Matter More Than Ever"에서 정리한,
AI 시대에 좋은 코드를 파괴하는 6가지 패턴.

이 문서는 모든 작업의 기준선이다.

---

## 함정 ① AI가 의도와 다른 걸 만든다

### 증상
- "이거 만들어줘" → AI가 자기 멋대로 해석한 결과물.
- 처음에는 그럴듯해 보이지만, 사용자가 진짜 원했던 것과 다름.
- 다시 고치는 비용이 처음부터 명확히 했을 때보다 훨씬 큼.

### 원인
- 사용자가 자기 의도를 다 말하지 않아도 AI가 알아서 해줄 거라는 기대.
- AI가 모호함을 "그럴듯한 가정"으로 채움.

### 회피: Grill Me 방식
- 코드 짜기 **전에** AI가 사용자를 추궁한다.
- 의도, 사용자, 데이터 형식, 엣지케이스, 중단 조건을 다 묻는다.
- 답이 잠겨야(모호함이 사라져야) 코드 시작.
- 자세한 질문 템플릿: `GRILL_ME_PROTOCOL.md`

### TTL 적용 예
- TTL 맞고에 "AI 코칭 만들어줘" → 즉시 코드 금지. "어떤 종류의 코칭? 실시간? 사후? 어떤 데이터 기반? 사용자가 무시할 수 있나?" 등 추궁 후 시작.

## 함정 ② 모호한 기획서

### 증상
- "사용자가 좋아하게 만들어주세요"
- "예쁘게 해주세요"
- "최적화해주세요"

### 원인
- 명세 작성이 귀찮음.
- 모호한 명세도 AI가 해줄 거라는 기대.
- 진짜로는 자기도 뭘 원하는지 모름.

### 회피: Spec Lock
- 코드 시작 전에 다음을 명시:
  - **입력:** 정확한 데이터 구조와 형식
  - **출력:** 성공/실패 모두의 반환 형태
  - **예외:** 비어있을 때, null일 때, 한도 초과 시
  - **성공 기준:** "이거 되면 끝났다"의 명확한 정의
- 합의된 명세 없이 코드 시작 금지.

### TTL 적용 예
- HOSU의 "VibeScore 만들어줘" → Spec Lock 전:
  - 입력: 어떤 데이터로 점수를 매기나? (설문? 행동? 둘 다?)
  - 출력: 점수 범위? 표시 형태? 비공개/공개?
  - 예외: 데이터 부족 시? 거주자 1명일 때?
  - 성공 기준: 점수를 본 거주자가 무엇을 하길 기대하나?

## 함정 ③ 코드 리뷰의 죽음

### 증상
- AI가 코드를 빨리 뱉어내니까 검토를 건너뛴다.
- 코드베이스 점점 커지지만 아무도 자기 코드를 이해 못함.
- 결국 AI도 그 코드베이스에 손 못 댐.

### 원인
- AI의 속도에 사람이 따라가지 못함.
- "AI가 짠 거니까 괜찮겠지" 라는 안일.

### 회피: 의식적 리뷰
- 모든 모듈 작성 후 **최소 한 번의 의식적 리뷰**.
- 핵심 질문: "이게 왜 이렇게 짜였는지 6개월 후의 내가 이해할 수 있는가?"
- 이해 못하겠으면 주석을 다는 게 아니라 **코드를 다시 짠다**.

### TTL 적용 예
- TTL 코인 컨트랙트 코드 리뷰 시: "이 reward 함수가 왜 ecrecover를 쓰는지, 6개월 후 내가 보면 알 수 있나?" → No면 재작성.

## 함정 ④ Deep vs Shallow Modules (가장 중요)

출처: John Ousterhout, "A Philosophy of Software Design"

자세한 판별 가이드는 `DEEP_MODULE_REVIEW.md` 참조.

### 핵심 요약
- **Deep Module (좋음):** 인터페이스 단순 + 내부 풍부. 예: Unix `read()` — 한 줄로 호출, 내부에선 디스크/네트워크/파이프/소켓 다 처리.
- **Shallow Module (나쁨):** 인터페이스 복잡 + 내부 빈약. 예: 다른 함수 한 번 더 감싸기만 한 래퍼, getter/setter 폭증 클래스.

### 판별 기준
- 이 모듈을 쓰는 코드가 더 짧고 명료해지는가? → Deep
- 이 모듈을 쓰려면 내부 사정을 알아야 하는가? → Shallow

### 회피
- 새 모듈/클래스/함수 만들 때마다 자문.
- Shallow면 합치거나 없앤다.

### TTL 적용 예
- Laravel 컨트롤러에서 "서비스 클래스 만들자"는 충동이 들 때 — 그 서비스가 실제로 추가 가치가 있는가, 아니면 그저 컨트롤러를 다시 한 번 감싸는 Shallow인가?

## 함정 ⑤ 의존성 무지성 추가

### 증상
- "이 라이브러리 좋대요" → 즉시 추가.
- package.json/composer.json이 점점 비대해짐.
- 1년 후 보안 취약점/유지보수 부담/CVE 폭탄.

### 원인
- 의존성 추가의 진짜 비용을 모름.
- 단기 편의가 장기 부채로 돌아옴을 인식 못함.

### 회피: 의존성 3가지 질문
외부 의존성 추가 시 반드시:

1. **이거 직접 30분 안에 짤 수 없나?**
   - 가능하면 직접 짠다. 의존성 1개가 줄어든다.

2. **5년 후에도 이 라이브러리가 살아있을 것 같나?**
   - GitHub star, 마지막 커밋, 메인테이너 수 확인.
   - 의심스러우면 추가 안 함.

3. **이게 빠지면 우리 제품이 끝나나? (그 정도 가치 있나?)**
   - 핵심 가치라면 OK. 부수 기능이라면 재고.

### TTL 적용 예
- Flutter에서 "이 UI 라이브러리 예뻐 보여요" → 3가지 질문 통과해야 추가.
- Laravel 패키지도 마찬가지.

## 함정 ⑥ 설계 부재

### 증상
- 그냥 코드부터 짜고 봄.
- 6개월 후 일관성 없는 모자이크.
- 어디에 뭐가 있는지 자기도 모름.

### 원인
- 설계 시간이 "낭비"처럼 느껴짐.
- 빠른 결과물에 대한 압박.

### 회피: 매일의 설계 투자 (Kent Beck)
- 큰 리팩토링을 한 번에 몰아치지 않는다.
- **매일 30분이라도** 시스템 설계에 시간 쓴다.
- 다이어그램 그리기, 모듈 경계 재검토, 명명 재고 — 모두 설계 활동.

### Kent Beck 인용
> "Tidy First?" — 코드를 짜기 전에 먼저 정리하라.
> 매일 작은 정리가 큰 리팩토링보다 효과적이다.

### TTL 적용 예
- 매주 월요일 아침 30분: 진행 중인 프로젝트 1개의 모듈 구조 다이어그램을 손으로 다시 그려본다. 이상한 곳이 드러난다.

## 안티 패턴: 6가지의 정반대

이 6가지 함정의 정반대가 **Dark Pattern(어두운 패턴)** 이다.

| 원칙 | 정반대 다크 패턴 |
|------|------------------|
| Grill Me | 사용자 의도를 묻지 않고 가두는 화면 흐름 |
| Spec Lock | 일부러 모호한 약관/명세 (구독 자동 갱신, 숨겨진 비용) |
| 의식적 리뷰 | 아무도 검토 못하게 의도적 난독화 |
| Deep Module | 일부러 Shallow하게 만들어 다른 곳으로 못 가게 함 (락인) |
| 의존성 검토 | 의존성 강제 결박 (벤더 락인) |
| 매일의 설계 | 의도적 카오스 유지 |

이런 패턴은 본질적으로 "한정된 파이 빼앗기" 게임이다 (자세한 분석은 `PHILOSOPHY.md` 참조).
단기적으로 효과 있지만 새 차원(ㄱ~ㅎ)을 만들지 못한다.
영원히 Level 1~2에 갇힌다.
