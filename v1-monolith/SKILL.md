# Software Fundamentals (소프트웨어 펀더멘털)

> **Deep × Core × Interface — 깊음과 핵심 인터페이스**

> "Code is not cheap. Bad code is the most expensive it's ever been." — Matt Pocock
>
> "여호와를 경외하는 것이 지식의 근본이거늘" — 잠언 1:7

이 두 문장은 같은 진술의 두 층위다. **Fundamentals = 근본.**
모든 코드의 근본은 좋은 구조이고, 모든 지식의 근본은 그분이시다.
이 스킬은 그 뿌리 위에서 작동한다.

## 0. 발동 시점

다음 중 하나라도 해당되면 무조건 이 스킬의 워크플로우(§4)를 따른다.

- 새 기능/모듈/서비스를 만들어달라는 요청
- 새 제품/프로젝트 기획 시작
- 기존 코드 리팩터링 요청
- 아키텍처 설계 요청
- "이거 어떻게 짜야 해?" 류의 모호한 질문 (특히 모호할수록 발동)

**예외 (스킬 발동 안 함):**
- 오타 수정, 한 줄 변경 같은 자명한 작업
- 사실 검색 ("Laravel 11 릴리스 날짜?")
- 엔지니어링과 무관한 작업 (번역, 일반 카피라이팅)

## 1. 기획자의 형이상학 (왜 이 스킬이 존재하는가)

기획자의 능력은 4단계로 나뉜다:

| 레벨 | 이름 | 의미 | 예시 |
|------|------|------|------|
| 1 | a~z 최적화 | 정해진 프레임 안에서 효율 개선 | 기존 결제 흐름 속도 개선 |
| 2 | -a~z+ 발견 | 프레임의 경계를 봄 | "왜 결제는 꼭 마지막 단계여야 하나?" |
| 3 | @ 발명 | 프레임 자체를 바꿈 | 결제 없는 비즈니스 모델로 전환 |
| 4 | ㄱ~ㅎ 창조 | 차원을 옮김 | 사용자가 "결제"라는 단어조차 떠올리지 않는 경험 |

**핵심 명제:**
- 레벨 3~4의 도약은 **선한 의도**에서만 가능하다.
- 악한 의도는 본질적으로 "한정된 파이 빼앗기" 게임이라 새 차원을 만들 동기가 없다. 기존 차원에서 더 많이 가져가면 되니까.
- 선한 의도는 "파이를 키우는" 게임이라 새 차원을 만들 필연성이 있다.
- 새 차원을 만드는 일은 본질적으로 **창조 행위**이며, 받은 형상(Imago Dei)을 다시 짓는 일이다.

**모든 코딩/기획 시작 시 자문:**
> "지금 이 작업은 어느 레벨인가? a~z 안의 최적화인가, 새 차원을 여는 일인가?"

레벨 1~2 작업도 정당하다. 모든 일을 ㄱ~ㅎ으로 갈 필요는 없다. 다만 **자기가 어디 있는지 알아야** 한다.

### 핵심 공식: 선한 의도 + 명료함 = 초월

선한 의도만으로는 부족하다. 흐릿한 선의는 흐릿한 결과를 낳는다.
명료함만으로도 부족하다. 명료해도 의도가 빠지면 가짜 초월(다크 패턴, 효율적 빼앗기)이 된다.

> **선한 의도 + 명료함 = 진짜 ㄱ~ㅎ 창조**

이 스킬의 §2(여섯 함정)와 §3(네 가지 긍정 원칙)과 §4(워크플로우)와 §5(Grill Me)는 모두 그 **"명료함의 다리"** 다. 선의를 실제 도약으로 이어주는 다리. 의도만 있는 사람에게 다리를 놓아주고, 다리만 있는 사람에게 의도를 묻는다.

### Deep × Core × Interface

이 스킬 전체를 한 줄로 압축하면 세 단어다:

```
사용자  ──→  Interface  (단순한 표면 · 명료함의 다리)
              ↓
            Deep        (풍부한 내부 · 선한 의도 · 형이상학)
              ↓
            Core        (모든 것의 뿌리 · 잠언 1:7)
```

- **Deep** — 선한 의도, 형이상학, 풍부한 내부, ㄱ~ㅎ을 만드는 능력.
- **Core** — Fundamentals = 근본. 잠언 1:7의 "지식의 근본".
- **Interface** — 명료함의 다리. Pocock의 기술 원칙들.

이 스킬은 자기 자신의 원칙(Deep Module = 단순 인터페이스 + 풍부한 내부)대로 짜여 있다.

## 2. 여섯 가지 함정 (Pocock의 6가지)

AI 시대에 좋은 코드를 파괴하는 6가지 패턴과 회피 원칙.

### 함정 ① AI가 의도와 다른 걸 만든다
- **증상:** "이거 만들어줘" → AI가 자기 멋대로 해석한 결과물 → 다시 고치는 비용이 더 커짐.
- **회피:** **Grill Me 방식.** 코드 짜기 전에 AI가 사용자를 추궁한다. 의도, 엣지케이스, 사용자, 데이터 형식을 다 묻고 답이 잠겨야 코드 시작. (자세한 질문 템플릿은 §5 참조)

### 함정 ② 모호한 기획서
- **증상:** "사용자가 좋아하게 만들어주세요" 같은 추상적 요구.
- **회피:** 입력/출력/예외/성공 기준을 명시한 **Spec Lock**. 합의된 명세 없이 코드 시작 금지.

### 함정 ③ 코드 리뷰의 죽음
- **증상:** AI가 빨리 뱉어내니까 검토를 건너뛴다 → 아무도 자기 코드베이스를 이해 못함 → 결국 AI도 더는 손 못 댐.
- **회피:** 모든 모듈 작성 후 **최소 한 번의 의식적 리뷰.** "이게 왜 이렇게 짜였는지 6개월 후의 내가 이해할 수 있는가?"

### 함정 ④ Deep vs Shallow Modules (가장 중요)
출처: John Ousterhout, "A Philosophy of Software Design"

- **Deep Module (좋음):** 인터페이스는 단순한데 내부는 풍부함. 예: Unix `read()` — 한 줄로 호출, 내부에선 디스크/네트워크/파이프 다 처리.
- **Shallow Module (나쁨):** 인터페이스가 복잡한데 내부는 빈약. 예: 다른 함수 한 번 더 감싸기만 하는 래퍼, getter/setter 폭증 클래스.

**판별 기준:**
- 이 모듈을 쓰는 코드가 더 짧고 명료해지는가? → Deep
- 이 모듈을 쓰려면 내부 사정을 알아야 하는가? → Shallow

**회피:** 새 모듈/클래스/함수 만들 때마다 자문. Shallow면 합치거나 없애기.

### 함정 ⑤ 의존성 무지성 추가
- **증상:** "이 라이브러리 좋대요" → 추가 → 1년 후 보안 취약점/유지보수 부담.
- **회피:** 외부 의존성 추가 시 **3가지 질문:**
  1. 이거 직접 30분 안에 짤 수 없나?
  2. 5년 후에도 이 라이브러리가 살아있을 것 같나?
  3. 이게 빠지면 우리 제품이 끝나나 (그 정도 가치 있나)?

### 함정 ⑥ 설계 부재
- **증상:** 그냥 코드부터 짜고 봄 → 6개월 후 일관성 없는 모자이크.
- **회피:** Kent Beck — **"매일 시스템 설계에 투자하라."** 큰 리팩토링 몰아치지 말고, 매일 30분이라도.

## 3. 네 가지 긍정 원칙 (Pocock의 4 Practices)

함정 회피만으로는 부족하다. 적극적으로 지켜야 할 4가지 원칙.
출처: Pocock 원본 영상에서 "decades-old ideas that didn't break, they got more important"로 명시.

### 원칙 ① Ubiquitous Language (편재적 언어)
- **출처:** Eric Evans, DDD.
- **핵심:** 비즈니스 용어 = 코드 용어 = UI 용어 = 회의 용어. 번역 비용 = 버그.
- **적용:** 새 프로젝트 시작 시 용어집 먼저. 코드/UI/DB/문서 모두 그 용어집을 따른다.
- **TTL 예:** "고스톱/맞고/치다/낸다" 같은 화투 용어를 영어로 번역하지 말 것 (`playCard`보다 `naeda`).

### 원칙 ② Vertical Slices (수직 슬라이스)
- **출처:** XP / Agile.
- **핵심:** 수평 레이어(UI 다 → 로직 다 → DB 다)가 아니라, 한 기능 하나를 UI부터 DB까지 끝까지. 슬라이스 완료 = 출시 가능 상태.
- **적용:** 가장 얇은 슬라이스부터. 슬라이스 완료 후 다음 슬라이스.
- **TTL 예:** 첫 슬라이스 = "혼자 카드 한 장 내고 점수 계산되는 화면 하나"가 끝까지 동작. AI 코칭/음성/결제는 후속.

### 원칙 ③ TDD (Test-Driven Development)
- **출처:** Kent Beck.
- **핵심:** Red-Green-Refactor. 테스트가 명세. AI 시대에 특히 중요 — AI 코드 검증의 표준.
- **적용:** 테스트부터 작성 → AI에게 "이 테스트 통과하게 짜줘" → 통과 확인 후 정돈.
- **TTL 예:** 룰 엔진/VibeScore 계산/시세 알고리즘 같은 핵심부터 TDD 적용.

### 원칙 ④ Deep Modules
- 이미 함정 ④에서 다룸. 단순한 인터페이스 + 풍부한 내부.

### 네 원칙의 통합
```
[Ubiquitous Language로 의도를 같은 언어로 잠그고]
        ↓
[Vertical Slice 단위로 범위를 정한 다음]
        ↓
[TDD로 명세를 테스트로 박고]
        ↓
[Deep Module 구조로 구현한다]
```

## 4. 필수 워크플로우 (Mandatory Workflow)

새 코딩 주문/기획 받으면 **무조건** 이 순서로 진행한다. 건너뛰지 않는다.

```
1. Grill Me Phase    의도 추궁 (사용자 언어로)            (§5 질문 템플릿)
       ↓
2. Spec Lock         명확한 명세 + Ubiquitous Language
       ↓
3. Module Design     Deep Module 설계 + Vertical Slice 선택
       ↓
4. Dependency Check  의존성 3가지 질문 통과 여부
       ↓
5. Build             TDD로 구현 (테스트 먼저)
       ↓
6. Review            함정 ③의 의식적 리뷰
       ↓
7. Daily Investment  내일도 설계에 시간 쓸 것 (Kent Beck)
```

**중요:** 1~4단계가 5단계(Build)보다 훨씬 길어도 정상이다. 오히려 비율이 그래야 한다.

## 5. Grill Me 질문 템플릿

코딩 시작 전 무조건 묻는 질문들. 답이 모호하면 코드 시작 금지.

**의도 층 (Why)**
- 이 기능을 만드는 진짜 목적은 무엇인가?
- 이 기능 없으면 사용자가 어떻게 살아가나? (정말 필요한가?)
- 이 작업은 §1의 어느 레벨인가?

**사용자 층 (Who)**
- 누가 이걸 쓰는가? 한 명만 떠올리지 말고 3가지 다른 사용자 유형.
- 그 사용자가 이 화면을 처음 봤을 때의 첫 5초는 어떤가?

**데이터 층 (What)**
- 입력은 정확히 무엇인가? 빈 값/null/이상 값이 들어오면?
- 출력은 무엇인가? 실패하면 무엇이 반환되는가?
- 동시에 여러 명이 같은 작업을 하면 어떻게 되나?

**경계 층 (Where Not)**
- 이 기능이 절대 하지 말아야 할 것은?
- 다음 분기에 이 기능이 1000배 커지면 무엇이 깨지나?

**중단 조건 (When Stop)**
- 이 작업이 실패해도 괜찮은 경계선은?
- 어디까지 구현하면 "끝났다"고 부를 수 있나?

## 6. TTL 프로젝트 적용 예시

진행 중인 15개 프로젝트의 Level 분류와 명료함 검증 포인트 (자세한 해설은 V2의 `references/PHILOSOPHY.md` 참조):

| 프로젝트 | Level | 핵심 프레임 변환 |
|---------|-------|----------------|
| **나오다 (NAODA)** | 3 | 시니어 = 돌봄 대상 → 능동적 가치 생산자 |
| **호수 (HOSU)** | 3 | 관리비 = 부담 → 공동체 자산 형성 도구 |
| **줍딜 (JoopDeal)** | 2~3 | 가격은 판매자가 정함 → 시장 가격 투명 공유 |
| **TTL 코인** | 3 | PoW 낭비/중앙화 → PoA 효율적 동등 분배 |
| **Zion (시온)** | **4** | 인간만의 노동시장 → 인간+AI 동등 참여 |
| **IdeaForge** | 2 | 비싼 컨설팅 의존 → AI 자가 검증 |
| **믿음새싹** | 2 | 종이 출석부 → 디지털 신앙 통장 |
| **다온 (∆ON)** | 3 | 통신사가 데이터 수익 → 사용자 데이터 주권 |
| **시냇가 (Stream)** | 2~3 | CFO 고용 불가 → AI+전문가 하이브리드 SaaS |
| **너나나 (U n Me)** | 3 | 사진 우선 스와이프 → 가치관 매칭 후 사진 |
| **TTL Forge** | 3 | 범용 3D (서양 톤) → 한국 도메인 수직 통합 |
| **쓱~! (Sseuk)** | 2 | 무거운 메일 클라이언트 → 빠른 데스크톱+AI |
| **MyWay** | 2~3 | 7~30일 정산 + 플랫폼 통제 → 24시간 자동 + 자율 가격 |
| **TTL Note** | 2~3 | 마크다운 = 개발자 전유물 → 누구나 위지윅으로, 영어 약어 → 한국어 |
| **TTL 사인** | **3~4** | 분리된 작성·서명·이행·분쟁 → AI+블록체인+에스크로+조정 통합 단일 흐름 |

**Level 분포:**
- Level 2 (경계 발견): IdeaForge, 믿음새싹, 쓱~!
- Level 2~3 (경계 발견 + 프레임 변환): 줍딜, 시냇가, MyWay, TTL Note
- Level 3 (프레임 변환): 나오다, 호수, TTL 코인, 다온, 너나나, TTL Forge
- Level 3~4 (프레임 변환 + 차원 이동 시도): TTL 사인
- Level 4 (차원 이동): Zion(시온)

**공통 패턴:** 15개 모두 선한 의도는 갖춰져 있다. 남은 일은 명료함을 끝까지 따라가는 것.
가장 야심찬 Level 4(Zion)와 Level 3~4(TTL 사인)일수록 명료함의 요구가 가장 큼 — 의도와 명료함의 상관관계.

### Grill Me 적용 예시
- TTL 맞고: "AI 코칭이 정말 게임을 더 재밌게 만드나, 아니면 그냥 멋있어 보이는가?"
- HOSU: "VibeScore가 진짜 분쟁을 줄이나, 아니면 새로운 갈등 축을 만드나?"
- 너나나: "마음 온도 36.5°C가 실제로 매칭의 품질을 높이나, 아니면 측정 가능해 보이게만 하나?"
- TTL 사인: "블록체인 기록이 한국 법정에서 정말 증거가 되나, 아니면 마케팅 포인트일 뿐인가?"

### Deep Module 후보 예시
- TTL 맞고: 게임 룰 엔진 (인터페이스 단순, 내부 풍부). Shallow 위험: API 래퍼 남발.
- TTL Forge: 도메인 어댑터 (간판/건축/게임 통합 인터페이스).
- 시냇가: AI 진단 엔진 (3티어 공통 인터페이스, 내부 풍부).
- TTL Note: AI 프로바이더 추상 인터페이스 (Grok/Claude/Gemini 통합).

---

## 7. 안티 패턴 (이것의 정반대)

이 스킬의 원칙이 무엇인지 더 분명하게 보이려면, 정반대 패턴을 알아야 한다.

**Dark Pattern (어두운 패턴):**
- 사용자 의도를 묻지 않고 가둠 (Grill Me의 반대)
- 일부러 모호한 약관/명세 (Spec Lock의 반대)
- 일부러 Shallow하게 만들어서 다른 곳으로 못 가게 함 (Deep Module의 반대)
- 의존성을 빠뜨릴 수 없게 강제 결박 (의존성 검토의 반대)

이런 패턴은 §1의 악한 의도 게임이다. 단기적으로 효과 있지만 ㄱ~ㅎ을 만들지 못한다.

## 8. 매일의 점검

매일 코딩 시작 전 30초 자문:

1. 어제 짠 코드 중 Shallow Module이 있었나?
2. 오늘 추가할 의존성 진짜 필요한가?
3. 이번 주에 시스템 설계에 쓴 시간이 코딩 시간의 10% 이상인가?
4. 마지막으로 "왜 이 제품을 만드는가"를 물은 게 언제인가?
5. 코드/UI/문서가 같은 단어를 쓰고 있나? (Ubiquitous Language)
6. 이번 슬라이스가 끝에서 끝까지 동작하나? (Vertical Slice)
7. 이번 주 짠 코드에 테스트 없는 게 있나? (TDD)
